Git 是版本控制工具，GitHub 是托管代码和协作的平台。初学者不必一开始理解 Git 的全部内部机制，但必须尽早学会最低可用路径：初始化仓库，查看状态，提交修改，推送到远程。因为学习计算机最重要的不是一次写出完美代码，而是让每次进步都有记录、可回退、可展示。

版本控制解决的是“变化管理”问题。没有 Git 时，你可能会保存 `final.docx`、`final2.docx`、`really-final.docx`。有 Git 后，每次重要变化都可以成为一次 commit。commit 不只是保存文件，也是给未来的自己写说明：我为什么做这个改动，它解决了什么问题。

最常见的四个命令是：

```bash
git status
git add .
git commit -m "Describe the change"
git push
```

`status` 看当前改了什么，`add` 把修改放入暂存区，`commit` 生成本地版本记录，`push` 把记录上传到远程仓库。刚开始不要追求复杂分支模型，先把这四步练熟。等你开始多人协作，再学习 branch、merge、pull request 和 conflict。

GitHub 的价值不只是存代码。它可以托管项目说明、记录 issue、展示贡献历史、发布静态网站。对个人学习者来说，一个维护良好的 GitHub 仓库就是作品集。哪怕项目很小，只要 README 清楚、代码能运行、提交历史真实，就比空泛地说“我学过 Python”更有说服力。

README 是仓库的门面。一个基本 README 至少包含：项目是做什么的，如何安装，如何运行，目录结构是什么，当前完成了什么，下一步计划是什么。初学者常常低估文档，其实文档是工程能力的一部分。别人能不能复现你的项目，首先取决于你有没有说明。

本章练习：创建一个名为 `computer-100-notes` 的 GitHub 仓库，把前几章学习笔记放进去。每章提交一次，commit message 写清楚，例如 `Add notes for file paths`。然后写一个 README，说明这个仓库是你的计算机入门学习记录。不要担心项目幼稚，早期作品的价值在于显示你的成长曲线。

常见误区有两个。第一，把 GitHub 当网盘，只上传最终文件，不保留过程。第二，害怕公开，结果永远没有可展示资产。公开并不意味着自夸，而是给学习建立一个严肃的外部约束。你会因为知道别人可能看到，而更愿意把内容整理清楚。
